Type-Safety ✗


function foo(n: number): string {
  return { foo: 'bar' }; // Error
}
foo(7);   // Ok
foo('a'); // Error

(transpiles to) 


function foo(n) {
  return { foo: 'bar' };
}
foo(7);   // returns '{ foo: 'bar' }'
foo('a'); // returns '{ foo: 'bar' }'

⇒ Type '{ foo: string; }' is not assignable to type 'string'.

⇒ Argument of type '"n"' is not assignable to parameter of type 'number'.

Though not type safe at run-time, TypeScript gives hints and warnings about incompatible types at compile-time.

Mehr als plain, typenloses JavaScript.

Nominal Typing ✗


interface Cat {
  color: string;
}

class Tiger implements Cat {
  color: string = 'striped';
}

// WATCH OUT (!!!): this is no compile error in TypeScript
let c: Tiger = { color: 'black' };

Comparable code in Java: Tiger t = (Tiger) new HashMap();

⇒ type checking is NOT enforced by name (class, interface)!

Vergleichbar: Tiger t = (Tiger) new Map<String, String>();

Compile Error in Java, C#, PHP(?), etc.

Structural Typing ✓


interface Cat {
  color: string;
}

class Tiger {    // Although 'Tiger' does not implement 'Cat'
  color: string; // 'Tiger' is kind of "a superset of" 'Cat'
}

let c: Cat;      // var 'c' is of type 'Cat'
c = new Tiger(); // 'Tiger' is type-compatible with 'Cat'

Comparable code in Java: Cat c = (Cat) new Tiger();

⇒ OK, because of structural typing!

https://www.typescriptlang.org/docs/handbook/type-compatibility.html

Vergleichbar: Cat c = (Tiger) new Map<String, String>();

Compile Error in Java, C#, PHP(?), etc.

Duck Typing ❀


interface Duck { name: string }
let ducktale: any = { name: 'Dagobert', money: 123456789 };
  // ↳ var 'ducktale' is an 'any' type

function doDuckTest(input: any): Duck {
  if (input && input.name && typeof input.name === 'string') {
    return (input as Duck); // DANGER ZONE unchecked type-cast
  }

  throw new TypeError('Not a Duck: ' + input);
}

let dagobert = doDuckTest(ducktale);
  // ↳ var 'dagobert' is a 'Duck' type - or TypeError thrown

⇒ NOT a TypeScript language feature

What kind of type?!?

TypeScript’s structural type system was designed based on how JavaScript code is typically written.

Because JavaScript widely uses anonymous objects like function expressions and object literals, it’s much more natural to represent relationships (..) with a structural type system instead of a nominal one.

Type Compatibility, TypeScript Handbook

Type Inference ✓


let x = 3;       // inferred type: 'number'
let foo = 'bar'; // inferred type: 'string'

foo = x + 4;     // 'number' (+) 'number' -> inferred 'number'

⇒ Type 'number' is not assignable to type 'string'.

Flow-Sensitive Typing ✓


function foo(input) {
  // A: 'input' can be an arbitrary type here
  if (typeof input === 'string') {
    // B: 'input' is supposed to be a 'String' type in this branch..
    let another: string = input.concat(' - es ist ein String!');
    // B: ..ends here
  } else if (typeof input === 'number') {
    // C: 'input' is supposed to be a 'Number' type in this branch..
    let calc: number = 8 * input;
    // C: ..ends here
  }
  // A: 'input' can be an arbitrary type here
}
